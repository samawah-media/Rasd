### 1. قرار التخزين: Supabase هو المسار الإنتاجي، والـ in-memory فقط fallback محلي

> [!WARNING]
> التوصيف الأدق للحالة الحالية: المنصة لم تعد تعمل بالكامل على **in-memory store**. يوجد `persistentStore` فعلي يستخدم Supabase عند توفر `SUPABASE_SERVICE_ROLE_KEY`، والسكيمة تحتوي أصلًا على `sources`, `source_rules`, `connector_runs`, `jobs`, و`captures` مع RLS. الخطر الحقيقي هو بناء أي جزء جديد من الـ scheduler/jobs/connectors فوق arrays محلية ثم نقله لاحقًا.

**القرار التنفيذي:**
- كل metadata/workflow جديد يجب أن يمر عبر `persistentStore` أولًا، مع fallback محلي للاختبارات فقط.
- `source_rules`, `jobs`, `connector_runs`, cursors, retries, usage events يجب أن تكون persistent من البداية.
- لا نبني scheduler/jobs على in-memory؛ هذا anti-pattern خطر لأن Vercel/Node runtime قابل لإعادة التشغيل في أي لحظة.
- لا نضيف طبقة تخزين جديدة من الصفر. نوسّع الموجود: Supabase Database للـ metadata، وSupabase Storage للأدلة والصور.

**تقسيم التخزين المطلوب:**
- **Database:** روابط، قواعد رصد، jobs، connector runs، cursors، حالات review، usage، dedupe hashes.
- **Storage:** thumbnails، screenshots، evidence cards، وأي visual proof نحتاج عرضه للعميل.
- **لا نخزن الفيديو كاملًا في MVP:** TikTok/Instagram video bytes مؤجلة حتى توجد حاجة قانونية/تشغيلية واضحة. نخزن permalink + metadata + thumbnail/screenshot فقط.

**مهمة تمهيدية قبل TikTok/Instagram:**
- [ ] تعديل `src/server/evidence-storage.ts` حتى يستخدم `organizationId` و`topicId` الحقيقيين بدل المسارات الثابتة `default/hidayathon`.
- [ ] إضافة اختبار يثبت أن مسار التخزين يحتوي organization/topic/item/capture الصحيح.
- [ ] التأكد أن `/api/captures/:id/asset` يبقى member-only ولا يكشف raw Supabase Storage path.

### 5. غياب `platform` من `MonitoringItem` type

[MonitoringItem](file:///d:/code%20-%20projects/RASD%20HAKSON/rasd-platform/src/lib/types.ts#L80-L104) لا يحتوي حقل `platform`. الحالي يعتمد على `sourceType` فقط لكنه لا يميز بين المنصات بشكل واضح في الواجهة.

**المقترح:** إضافة حقل `platform` اختياري:
```ts
platform?: 'tiktok' | 'instagram' | 'x' | 'web' | 'rss' | 'unknown';
```

### 6. `ReportItemCard.platform` لا يدعم TikTok/Instagram

[ReportItemCard](file:///d:/code%20-%20projects/RASD%20HAKSON/rasd-platform/src/lib/types.ts#L116-L134) يحتوي:
```ts
platform: "X" | "Website" | "News" | "Official";
```
**يجب توسيعه** ليشمل `"TikTok" | "Instagram"`.

---

### 7. Python Dependencies غير مذكورة بوضوح

الخطة تذكر `instaloader` و `TikTok-Api` لكن لا تحدد:
- هل سنستخدم `venv` أم `uv` أم Docker؟
- هل سنضع Python داخل Vercel runtime أم worker خارجي؟ حتى لو كان runtime متاحًا، scraping طويل أو dependencies ثقيلة ليست خيارًا مناسبًا كمسار أساسي.

**الحل:** لا يدخل Python/instaloader في MVP. نبدأ بـ Node + OpenGraph/oEmbed، ونجعل `yt-dlp` provider اختياريًا لا يكسر الحفظ. إن احتجنا Python لاحقًا يكون worker منفصل أو provider lab-only.
